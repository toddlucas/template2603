**Date:** 2026-02-17  
**Status:** Implemented (MVP POC)

## Decisions

| Question | Decision |
|----------|----------|
| Streaming? | Yes — SSE (`text/event-stream`) |
| Multi-turn context? | Yes — caller owns history, sends full `messages[]` each turn |
| LLM provider? | OpenAI (GPT-4o) via `Microsoft.Extensions.AI` |
| Project placement? | `Base2.Services` (no new project) |

## API Shape

```
POST /api/projects/{projectId}/chat/turns
Content-Type: application/json
Authorization: Bearer {token}
```

**Request body** (`ChatTurnRequest`):
```json
{
  "sessionId": "guid",
  "messages": [
    { "role": "user", "content": "Rewrite section 3 to be more concise." }
  ],
  "documentIds": ["guid", "guid"]
}
```

**Response** — `text/event-stream`:
```
event: text
data: {"content":"Here is a revised version..."}

event: text
data: {"content":" of section 3."}

event: done
data: {"changesets":[{"id":"guid","status":"Pending",...}]}
```

The `done` event is always last and carries any changesets created by tool calls.

## Tool Calls (LLM-facing)

| Tool | Parameters | Delegates to |
|------|-----------|--------------|
| `list_documents` | `(projectId)` | `DocumentService.ListByProjectAsync` |
| `read_document` | `(documentId)` | `DocumentService.GetMarkdownAsync` |
| `propose_edit` | `(documentId, description, editedMarkdown)` | `ChangesetService.CreateAsync` |
| `get_changeset` | `(changesetId)` | `ChangesetService.GetAsync` |

Tool invocation is handled automatically by M.E.AI's `UseFunctionInvocation()` middleware.

## Configuration

Add to `appsettings.json` / user secrets:

```json
{
  "ChatClient": {
    "ApiKey": "sk-...",
    "Model": "gpt-4o"
  }
}
```

When `ApiKey` is empty, `IChatClient` is not registered. Tests supply a stub via DI override.

---

## Multi-Turn Context

The client maintains the conversation history and sends the full `messages[]` array on every
request. The server prepends a system prompt and any requested document context, then appends
the caller's history before sending to the LLM. No server-side session storage is required.

## Tests

### `Base2.Services.Test` — Unit

- **No tool calls** — plain text reply yields text chunks + done chunk with empty changesets.
- **Empty documentIds** — succeeds without document context.
- **Multi-message history** — full history passed to `IChatClient`.
- **Done chunk is last** — invariant verified.

### `Base2.Web.Test` — Integration

- **Unauthenticated** → 401.
- **Authenticated, stub client** → 200 `text/event-stream` with `event: text` and `event: done`.
- **Null body** → 400.

## Known Limitations / Next Steps

- `ChatService` does not yet persist the assistant reply to a `ChatSession` entity. For the MVP
  the client is responsible for maintaining history.
- The `propose_edit` tool collects changesets in a local list inside `StreamTurnAsync` but the
  list is not yet wired back through the tool closures — a follow-up is needed to thread the
  `createdChangesets` collection into the tool lambda.
- Rate limiting on the chat endpoint should be tightened (token-based, not request-based) before
  production use.
